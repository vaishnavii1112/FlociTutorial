# floci-gcp for Python: Store and Retrieve Application Secrets on Google Cloud Secret Manager Locally with Versions, Rotation, and Lifecycle Control

In this tutorial, we build a complete Google Cloud Secret Manager workflow that runs entirely on our own machine using **floci-gcp**, a free, open-source, drop-in local Google Cloud emulator that speaks the real GCP gRPC protocol on `localhost:4588`. We begin by starting a floci-gcp container and pointing the official `google-cloud-secret-manager` client at the emulator using an environment-variable switch, so the exact same code runs against real GCP by removing that variable. We then create a secret container, add a secret version holding the actual sensitive bytes, and access the payload back through the `versions/latest` alias. We rotate the secret by adding a new version, list all versions and secrets, and use lifecycle controls to disable, re-enable, and permanently destroy specific versions. Finally, we delete the secret entirely, closing the loop on a full store-and-retrieve secrets lifecycle, all without a Google Cloud account, a service-account key, or a single cent of cloud spend.

Because floci-gcp implements the same gRPC responses that the GCP client libraries expect, everything you write here transfers directly to real Secret Manager by pointing the client at Google instead of the emulator.

## Starting floci-gcp and Connecting the Secret Manager Client

```bash
# Start the local GCP emulator (all services share port 4588)
docker run -d --name floci-gcp -p 4588:4588 floci/floci-gcp:latest

# Confirm it is healthy
curl http://localhost:4588/health

# Install the official Secret Manager client
pip install google-cloud-secret-manager
```

```python
import os
import time

# IMPORTANT: set the emulator host BEFORE importing/creating the client.
os.environ["SECRET_MANAGER_EMULATOR_HOST"] = "localhost:4588"
os.environ.setdefault("GOOGLE_CLOUD_PROJECT", "floci-local")

from google.cloud import secretmanager
from google.api_core.exceptions import AlreadyExists

PROJECT = "floci-local"
PARENT = f"projects/{PROJECT}"
SECRET_ID = "db-password"

# Unlike the Pub/Sub and Firestore clients, the Secret Manager client does not
# automatically route to SECRET_MANAGER_EMULATOR_HOST, so when that variable is
# set we point the client at the emulator over a plaintext gRPC channel. With the
# variable unset (as on real GCP), the client uses its normal secure endpoint and
# your default credentials, so the same code runs against the cloud unchanged.
EMULATOR_HOST = os.environ.get("SECRET_MANAGER_EMULATOR_HOST")
if EMULATOR_HOST:
    import grpc
    from google.cloud.secretmanager_v1.services.secret_manager_service.transports import (
        SecretManagerServiceGrpcTransport,
    )
    transport = SecretManagerServiceGrpcTransport(
        channel=grpc.insecure_channel(EMULATOR_HOST)
    )
    client = secretmanager.SecretManagerServiceClient(transport=transport)
else:
    client = secretmanager.SecretManagerServiceClient()

def banner(title: str):
    print("\n" + "=" * 68)
    print(title)
    print("=" * 68)

def wait_for_floci(timeout: int = 60):
    deadline = time.time() + timeout
    last = None
    while time.time() < deadline:
        try:
            list(client.list_secrets(request={"parent": PARENT}))
            print("floci-gcp Secret Manager is ready at",
                  os.environ["SECRET_MANAGER_EMULATOR_HOST"])
            return
        except Exception as e:
            last = e
            time.sleep(1)
    raise RuntimeError(f"floci-gcp did not become ready: {last}")

wait_for_floci()
```

We begin by launching floci-gcp with a single `docker run` command that exposes port `4588`, then set `SECRET_MANAGER_EMULATOR_HOST` before creating the `google-cloud-secret-manager` client. Unlike the Pub/Sub and Firestore clients, the Secret Manager client does not automatically pick up that variable, so when it is present we build the client on a plaintext gRPC channel aimed at the emulator; with the variable unset (on real GCP) the client falls back to its normal secure endpoint and your default credentials, so the code transfers unchanged and no service-account key is needed locally. Secret Manager identifies resources by resource-name paths, so we precompute the project `PARENT` (`projects/floci-local`) and poll `list_secrets` until the emulator answers.

## Creating a Secret and Adding Its First Version

```python
banner("Create secret + add version")

def ensure_secret():
    try:
        secret = client.create_secret(
            request={
                "parent": PARENT,
                "secret_id": SECRET_ID,
                "secret": {"replication": {"automatic": {}}},
            }
        )
        print("Created secret:", secret.name)
    except AlreadyExists:
        secret = client.get_secret(request={"name": f"{PARENT}/secrets/{SECRET_ID}"})
        print("Secret already exists:", secret.name)
    return secret

secret = ensure_secret()

# The secret container holds no data by itself; the bytes live in a version.
v1 = client.add_secret_version(
    request={"parent": secret.name, "payload": {"data": b"initial-P@ssw0rd"}}
)
print("Added version:", v1.name)
```

We create the secret container with `create_secret`, specifying automatic replication (the standard policy) and catching `AlreadyExists` so the script is idempotent. Crucially, a secret in Secret Manager is just a named container with metadata; the actual sensitive bytes live in immutable **versions**. So we call `add_secret_version` with the raw payload `data` to store the first value, which returns the version's resource name ending in `/versions/1`.

## Accessing Secret Payloads

```python
banner("Access secret payloads")

# Access the newest value through the 'latest' alias.
resp = client.access_secret_version(
    request={"name": f"{secret.name}/versions/latest"}
)
print("latest value:", resp.payload.data.decode("utf-8"))

# Access a specific, pinned version by number.
resp_v1 = client.access_secret_version(request={"name": v1.name})
print("version 1 value:", resp_v1.payload.data.decode("utf-8"))
```

We retrieve secret values with `access_secret_version`. Using the `versions/latest` alias always returns the most recently added, enabled version, which is what applications typically read at startup. We also access a specific version by its full resource name to show that historical versions remain individually addressable. The payload comes back as raw bytes in `resp.payload.data`, which we decode to a string; storing bytes means Secret Manager works equally well for text passwords and binary key material.

## Rotating, Listing, and Lifecycle Control

```python
banner("Rotate + list + lifecycle")

# Rotate: add a new version. 'latest' now points at v2.
v2 = client.add_secret_version(
    request={"parent": secret.name, "payload": {"data": b"rotated-N3wP@ss"}}
)
print("Added version:", v2.name)
print("latest is now:",
      client.access_secret_version(
          request={"name": f"{secret.name}/versions/latest"}
      ).payload.data.decode("utf-8"))

# List all versions of the secret.
print("\nAll versions:")
for v in client.list_secret_versions(request={"parent": secret.name}):
    print(f"  {v.name.split('/')[-1]:>3}  state={v.state.name}")

# Disable the old version so it can no longer be accessed.
client.disable_secret_version(request={"name": v1.name})
print("Disabled version 1")

# Re-enable it, then permanently destroy it (irreversible; payload is wiped).
client.enable_secret_version(request={"name": v1.name})
client.destroy_secret_version(request={"name": v1.name})
print("Destroyed version 1")

# List all secrets in the project.
print("\nSecrets in project:")
for s in client.list_secrets(request={"parent": PARENT}):
    print("  ", s.name)
```

We rotate the secret by adding a second version, after which the `versions/latest` alias automatically resolves to the new value, the mechanism apps use to pick up rotated credentials without code changes. We enumerate versions with `list_secret_versions`, printing each version's `state` (ENABLED, DISABLED, or DESTROYED). We then walk the full lifecycle on version 1: `disable_secret_version` makes it inaccessible while retaining the data, `enable_secret_version` restores access, and `destroy_secret_version` permanently and irreversibly wipes the payload while keeping the version's metadata. Finally we list all secrets in the project.

## Cleaning Up

```python
banner("Cleanup")

client.delete_secret(request={"name": secret.name})
print(f"Deleted secret: {secret.name}")

remaining = [s.name for s in client.list_secrets(request={"parent": PARENT})]
print("Secrets remaining:", remaining)
print("Secret Manager workflow complete.")
```

We finish by deleting the entire secret with `delete_secret`, which removes the container and all of its versions in one call, then re-list the project's secrets to confirm it is gone. This leaves the emulator in a clean state.

## Conclusion

In conclusion, we have a complete Google Cloud Secret Manager workflow that runs locally on floci-gcp and transfers to real GCP by simply not pointing at the emulator. We started a credential-free emulator, connected the official `google-cloud-secret-manager` client with one environment variable, and exercised the operations that real applications depend on: creating a secret container; adding versions that hold the sensitive bytes; accessing values through the `latest` alias and by pinned version; rotating by adding new versions; listing versions and secrets; and driving the full lifecycle with disable, enable, and destroy before deleting the secret. Because floci-gcp mirrors the Secret Manager gRPC protocol, every call we made here is production-accurate, which means we can build and test entire secret-management flows fast, offline, and for free before running the same code against the cloud.
